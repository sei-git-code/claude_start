# ComfyUI × 動画AI × Codex スプライト量産パイプライン（Claude Code設定ではない、技術ノウハウの記録）

出典: [ComfyUI × 動画生成AI × Codex で、ゲーム用スプライトアニメを量産するパイプラインを作った話](https://qiita.com/archeleeds/items/2efad73069b54288deb4)

注意: ここでの「Codex」はOpenAI/ChatGPT側のコーディングエージェントCLIであり、Claude Codeではない。導入対象ではなく、将来ゲーム用ドット絵アセットを生成AIで作る際の参考として記録。

## パイプライン
```
キャラシート生成 → 画風統一 → LoRA学習(kohya_ss) → ポーズ型・ドット基準画像の用意
  Step1 CN     : ポーズ型+depth ControlNetで立ち絵候補を生成
  Step2 SELECT : 人間が採用1枚を選ぶ
  Step3 I2V    : Wan2.2 I2V-A14Bで採用画像からモーション動画を生成(49フレーム)
  Step4 SELECT : 動画から採用フレームを選ぶ(8〜12枚)
  Step5 DOT    : Codex CLIの画像生成ツールでドット絵変換 → 透過/1:1正規化/クロップ
```

## 転用可能な設計原則
1. 同一性はLoRA、構造はControlNet、変化はプロンプト、一貫性はリファレンス画像 — 手段を混ぜない
2. ポーズの正解とデザインの正解を分離して指示する（"STRICT ROLE SPLIT"）
3. 生成AIの出力は機械補正で締める（サイズはbbox高さ比、位置はbbox中心の平行移動）
4. 数値検証は必要条件、最終判定は目視
5. 非決定的で高価な工程と、決定的で反復したい工程を分離する（ハッシュベースの永続キャッシュ）
6. 各工程は自分名義のフォルダにのみ書き込む
7. 品質保証は前工程へ（最小枠クロップで不良が見た目で分かる形式にする）
8. 正本はスクリプト、UIは使い捨て

## ハマりどころ
- ControlNetで新規ポーズ型は作れない（循環問題）→ プロンプト駆動txt2imgか動画AIに丸投げ
- 動画AI(I2V)はstart-onlyが基本。始点終点を両方固定すると終点品質が全体を支配する
- ドット絵化はSDXL系ピクセルアートLoRAでは実用品質に届かない
- グリッドピッチ推定は自己相関のピークが倍音に誤ロックしやすい
- 透過はAIセグメンテーションより「画像の縁に連結した領域だけを背景とする」境界連結判定+色キーが安定
